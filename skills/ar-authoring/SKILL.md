# 应用级需求 / 用户故事（AR）撰写

把一个功能点拆成**可验收、可追溯、面向非技术干系人也读得懂**的用户故事（AR）。每条 AR = 一句 User Story（WHAT + WHY）+ 一组 Given-When-Then 验收标准（AC）+ 优先级。**不写 HOW**（技术栈 / 接口 / schema 全部下沉详细设计）。

## 权威规范（必读）

完整方法论、拆分粒度规则、字段规范、GWT 写法、单条模板、自检清单都在 **`references/AR需求撰写规范模板-v4.md`**。它是本 skill 的唯一结构权威——动笔前 Read 它。本文件只做导航与要点提炼。

规范要点定位：
- **§一 核心方法论** —— 拆分单元是 User Story（不是模块/页面/接口）；只写 WHAT/WHY；强制优先级。
- **§二 拆分粒度** —— 何时拆条、何时合并为多场景（避免过度拆分）。
- **§三 字段规范** —— 名称、描述（`作为…我希望…以便于…` 单句）、**验收标准（§3.3，工作量最大最易错）**：GWT 三段、阿拉伯数字编号、字段展示写进 Then、§3.3.5 必须覆盖的常见场景类型。
- **§四 单条模板** —— 可直接复制填写。
- **§六 撰写自检清单** —— 28 项 / 6 类（内容质量 / 拆分粒度 / 验收标准 / 假设 / 不确定项 / 元数据），提交前逐条核。

## 工作流

1. **定单元**：按规范 §一/§二 把功能切成 User Story 粒度——同一动作的多种正常分支合并为多场景，跨模块联动单独成条，不同权限/技术栈/CRUD 各拆。
2. **写描述**：每条用 `作为<who>，我希望<what>，以便于<why>` 单句；名称 ≤ 标准长度、不含实现细节。
3. **写验收标准（重头）**：每个场景 `【Given】<前置>【When】<触发>【Then】<期望>`，数字编号；**字段展示写进 Then**；**逐条检查 §3.3.5 常见场景类型**——空数据态、CRUD 列表反馈、分页、排序、面包屑、按钮置灰、二次确认，以及**边界 / 异常 / 越权 / 环境异常**都用 GWT 写入（不另立「边界场景」节）。时间/数量/阈值给**具体数值**。
4. **补优先级 + 假设 + 不确定项**：优先级带理由；假设显式列出；`[NEEDS CLARIFICATION]` ≤ 3 个，评审版清零。
5. **自检**：跑 §六 全 28 项。

## 输出

一条或一组 AR 需求卡（按规范 §四 模板），含：名称 / User Story 描述 / 优先级（+理由）/ 验收标准（GWT）/ 假设 / 不确定项 / 元数据（责任人、验收人、编号）。

## 自检红线（提交评审前 · 摘自 §六）

- 每条 AC 用【Given】【When】【Then】（或合法简写，但 Then 必有）。
- **边界 / 异常 / 越权 / 环境异常已用 GWT 覆盖**（最常见漏项）。
- 全文 WHAT/WHY，无技术栈 / 框架 / API 选型。
- 阈值/数量/时间给具体值；`[NEEDS CLARIFICATION]` 评审版已清零。
- 责任人 / 验收人 / 编号不为空且格式正确。

## 可选 · 项目流程集成

> 若你的流程没有迭代 / 版本概念，跳过本节。

若像 KDevSec SOP 那样区分 迭代 / 版本：
- **AR 是版本级产物**，归档到版本目录（如 `迭代N/vX-<版本名>/01-ar/AR需求.md`）。
- 头部加 front-matter：`迭代 / 版本 / 类型: AR / 状态 / 版本号 / 上游依据(@版本号，钉 SR 与故事地图) / 日期`。
- AR 与高保真原型**配对联合评审**：原型每个交互点可追溯到 ≥1 条 User Story；冻结后单独改任一件须联合复核。
- 平台级横切交互约定（权限可见性、置灰气泡、确认文案、长任务异步等）若已沉淀为单独文档，则各卡直接引用、不逐卡重复。
