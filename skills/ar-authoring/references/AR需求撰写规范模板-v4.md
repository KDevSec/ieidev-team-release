# 需求撰写规范模板

> 本规范融合两套规则：
> - 方法论骨架：[需求拆分规则模板.md](需求拆分规则模板.md)（来自 speckit `/speckit-specify` 命令）
> - 落地细则：[Mode2-需求撰写规则.md](Mode2-需求撰写规则.md)（来自 KSecure 项目日常需求撰写经验）
>
> 适用范围：迭代型企业项目的 Sprint Backlog 需求条目撰写、需求评审、需求拆分。
>
> 版本说明：v2，依据评审意见调整——边界/异常场景与字段定义全部统一融入"验收标准"，不再独立成节。

## 一、核心方法论

### 1.1 拆分单元：User Story（用户故事），不是模块/页面/接口

每个 User Story 必须满足 **"四独立"**：

- **独立可测**（Independent Test）
- **独立可开发**
- **独立可部署**
- **独立可演示**

衡量标准：**砍掉任意低优先级 Story，只实现最高优先级 Story，仍然是一个能向用户交付价值的 MVP**。

### 1.2 关注 WHAT 与 WHY，不写 HOW

- ✅ 写：用户能做什么 / 为什么需要 / 达成什么效果 / 实际生效行为
- ❌ 不写：技术栈选型、框架、API 设计、数据库结构、代码结构

实现细节归入"技术方案"阶段，不进入需求文档。

### 1.3 强制优先级

| 优先级 | 含义 | 判定标准 |
|--------|------|----------|
| 高（P1） | 核心价值 / MVP / 阻塞 | 客户必须、影响主流程、缺失则功能不成立 |
| 中（P2） | 重要补强 | 重要但可延后；编辑/优化类、主分支补全 |
| 低（P3） | 锦上添花 | 长尾场景、便利性增强 |

## 二、需求拆分粒度规则

### 2.1 必须拆条的场景

| 拆分维度 | 拆分示例 |
|---------|---------|
| **不同技术栈** | 同一功能跨技术栈 → 拆成 `【docker】查看容器列表` 与 `【podman】查看容器列表` |
| **不同操作权限 / 动作** | 同一对象的"允许操作"和"禁止操作"独立成条 |
| **CRUD 各为独立需求** | 列表、添加、编辑、删除、关联各拆一条 |
| **跨模块联动** | 日志上报、防御模式切换等单独成条 |
| **不同优先级** | 同一功能内若部分 P1 / 部分 P3，按优先级拆开 |

### 2.2 不拆条、合并为同一条的场景

- 同一动作的**多种正常分支**（如多种允许权限）→ 合并为 1 条，通过验收标准里 `1、2、3、…` 多场景体现
- **弹框中的所有字段**在同一动作内完成 → 描述写"输入策略名称、描述、文件保护规则…"，**不为字段单独拆条**

## 三、需求字段规范

### 3.1 需求名称

- **句式**：`【范围/技术】+ 动宾短语`
- **跨技术栈** 用方括号区分：`【docker】查看容器列表信息`、`【podman】查看容器列表信息`
- **通用功能**不带方括号：`编辑策略文件防护规则`、`关联策略模版`、`设置防御模式`
- **动词建议**：查看 / 添加 / 编辑 / 删除 / 设置 / 关联 / 配置
- **长度** ≤ 20 字
- ❌ 名称中**不写实现细节**（实现细节进描述 / 验收标准）

### 3.2 需求描述（User Story 用户故事）

固定句式：

```
作为<角色>，我希望<功能/能力>，以便于<目的/价值>
```

示例：

> 作为运维人员，我希望可以编辑策略中的文件防护规则，以便于修改规则的信息

规则：

- **角色**：项目内统一约定一个主角色（如"运维人员"）；如有其他角色，需在项目级文档中显式约定
- **功能**：动宾短语，与需求名称语义对齐
- **目的**：必须是**业务/管理价值**，不是技术实现
- 一条需求只写**一句用户故事**，不罗列多角色多目的

### 3.3 验收标准（最重要、工作量最大）

> 这是整份需求文档中工作量最大、最容易出错的字段。**所有内容统一在这里表达**——包括正常场景、字段展示、边界/异常、联动、策略生效——都以 GWT 格式呈现，不再单独立"边界场景"或"关键实体"小节。

#### 3.3.1 总体结构

- 用阿拉伯数字编号 `1、2、3、…` 列出**所有验收场景**
- 每个场景一行，**场景标题**后用全角 `：` 引出 GWT 内容
- 场景之间用换行分隔

#### 3.3.2 GWT（Given-When-Then）写法

**强制使用中文方括号标记三段**：

```
【Given】<前置条件>【When】<触发动作>【Then】<期望结果>
```

示例：

```
【Given】主机内已安装 docker
【When】通过 docker 部署容器并刷新容器列表
【Then】容器列表中正确展示容器名称、容器镜像、运行状态…
```

规则：

- 三段必须**同时出现**且顺序固定
- 简写允许：当前置/触发显而易见时可省略 Given，只写 When/Then，但 **Then 必须有**
- 单一场景验收标准允许只写一段 GWT，无需编号

#### 3.3.3 多结果分点写法

Then 包含多个结果时，用**全角圆括号**编号：

```
【Then】
（1）容器列表中策略名称显示设置的策略名称
（2）主机容器中…只允许读/执行操作
（3）主机容器中使用非…无任何操作权限
```

❌ 不要用 `(1)`、`1)`、`①` 等其他编号格式。

#### 3.3.4 字段展示写入验收标准  

> 功能涉及的关键实体、字段、属性，**不要单独列"关键实体"**，应直接写在 Then 中作为可观察结果。列表中的字段信息一行展示。

示例：

```
【When】打开容器列表页
【Then】列表中每行展示以下字段：
容器名称，容器镜像（含镜像名 + tag），运行状态（运行中、已暂停、已停止），防护策略（已关联策略名 / 未关联时显示「—」），创建时间（YYYY-MM-DD HH:mm:ss）
```

字段的取值范围、格式、默认值、空值表现都在此体现。

#### 3.3.5 必须覆盖的常见场景类型

每条需求都要检查以下类型，对应场景必须在验收标准中以 GWT 形式体现：

| 场景类别 | 常见验收要点 |
|---------|--------------|
| **字段展示** | 列表/详情页所有字段的展示内容、格式、默认值、空值 |
| **空数据态** | "暂无数据"提示 |
| **新增/删除/编辑后的列表反馈** | 列表是否正确展示 / 不再显示 |
| **状态变化/数据刷新** | 同步周期（如"最长 250 秒刷新"） |
| **按钮置灰/置亮** | 未勾选 / 勾选数量不同 → 按钮状态 |
| **列表分页** | 默认 10 条/页 |
| **列表默认排序** | 如"按创建时间降序" |
| **页面面包屑** | 如"安全防御 / 主动防御 / 容器防护" |
| **二次确认** | 删除等危险操作弹窗确认 |
| **覆盖/转换提示** | 如"是否继续？"模态提示 |
| **策略生效结果** | 容器中**实际行为**（允许 / 拦截）是否符合预期 |
| **联动模块** | 如日志上报、搜索条件增加 |
| **边界条件** | 数据为空、达到上限、跨边界值 |
| **异常输入 / 越权** | 非法字符、未授权用户、参数缺失 |
| **环境异常** | 网络中断、并发冲突、超时、降级 |

#### 3.3.6 用语与符号约定

| 项目 | 约定 |
|------|------|
| **括号** | 中文场景一律使用全角 `（）`、`【】`、`「」`；英文专名/技术名（docker、podman、CPU）保留半角 |
| **顿号** | 并列名词用 `、`，不用 `,` |
| **冒号** | 场景标题与详细描述之间用全角 `：` |
| **强调** | 不使用任何 Markdown 标记（纯文本单元格）；强调用 `「」` 或括号说明 |
| **数值/时间** | 写明具体阈值（`最长 250 秒`、`默认一页 10 条`、`3 次失败锁定 60 秒`），❌ 禁止"较短时间""一定时间内"等模糊表述 |
| **路径/枚举值** | 用 `（）` 列举，例如 `运行状态（运行中、已暂停、已停止）` |
| **跨模块关联** | 必须显式说明，如"防护类型中**增加**容器防护搜索条件" |

### 3.4 假设与依赖（Assumptions）

列出"用户没明说、按行业默认值假设了"的内容，让评审能确认或推翻：

- 目标用户假设：例如"用户具备稳定的网络连接"
- 范围边界假设：例如"v1 不含移动端原生应用"
- 数据 / 环境假设：例如"复用现有的用户认证系统"
- 外部依赖假设：例如"依赖现有的用户档案 API"
- 合规假设：例如"按 GDPR 标准保留数据 30 天"

### 3.5 不确定项（最多 3 个）

撰写过程中遇到无法确认的关键点，用 `[NEEDS CLARIFICATION: 具体问题]` 标记。

- **全文最多 3 个**
- 优先级顺序：**范围 > 安全/隐私 > 用户体验 > 技术细节**
- 其余按行业默认值假设，并写入 Assumptions
- 评审时需逐条澄清并清零

## 七、反模式速查（常见错误 → 修正方式）

| 反模式 | 修正方式 |
|--------|----------|
| 按"登录页 / 注册页 / 个人中心"拆 | 改成"新用户首次登录并完成首要任务"这种端到端旅程 |
| P1 / P2 / P3 都得做完才有价值 | P1 没切对——重新找最小可演示切片 |
| 需求名称里写"使用 Redis 缓存容器列表" | 删掉技术词，改为"查看容器列表信息" |
| 验收标准里写"系统应快速响应" | 改为"95% 请求 1 秒内出结果"或具体阈值 |
| 多结果用 `(1)(2)(3)` 或 `1) 2) 3)` | 改为全角 `（1）（2）（3）` |
| Then 中只写界面变化，没写端侧实际生效 | 涉及策略下发的需求必须补充端侧行为验证 |
| 弹框里 5 个字段拆成 5 条需求 | 合并为 1 条，描述写"输入字段 A、B、C、D、E" |
| docker / podman 同一功能合写一条 | 拆成 `【docker】xxx` 与 `【podman】xxx` 两条 |
| 全文 10 个 `[NEEDS CLARIFICATION]` | 按 范围 > 安全 > UX > tech 排序，保留 3 个最关键的，其余按行业默认值假设并写入 Assumptions |
| 描述里写两个角色三个目的 | 一条需求只写一句用户故事，多角色多目的拆条 |
| 把"边界与异常场景"单独成节 | 用 GWT 写入验收标准，作为编号场景之一 |
| 用"关键实体 / Key Entities"列字段 | 字段直接写进 Then 的多结果分点中，作为可观察展示项 |
