# 顶层约束 / 宪法生成

为项目产出**最高层执行准则**根文档（如 `CLAUDE.md`）：声明优先级、硬约束、技术语境引用、流程引用，并设计**触发路由表**（任务特征 → 必读子约束）与**冲突裁决链**。目标是让任何 AI Agent 动手前加载一份文件，就知道「该读什么、谁说了算、冲突怎么裁」。

## 权威规范（必读）

完整的两相位流程、骨架、各节小模板、自检都在 **`references/top-constraint-spec.md`**。它是本 skill 的唯一结构权威——动笔前 Read 它，按它的两相位走。本文件只做导航。

规范要点定位：
- **§1 输出文档通用骨架** —— 最小骨架 + 可选挂载段落 + 三种典型形态（按项目复杂度组合）。
- **§2 前置材料目录** —— 6 类标准 prereq（C1-C6）与项目专有变量。
- **§3 Phase 1 · 探测 + 设计** —— Step0 现存根约束检测 → Step1 C1-C6 用户清单探测（**用户清单优先**）→ Step2 挂载决策 → Step3 派生段落草稿 → Step4 **设计报告 + 评审 checkpoint**。
- **§4 Phase 2 · 生成** —— 装配顺序 → 各节双子块小模板 → 写文件 → 内联自检。
- **§5 推荐加载验证**、**§6 走查示例**。

## 工作流（两相位，中间有人工 checkpoint）

**Phase 1 · 探测 + 设计**（先设计、后生成，不要跳过）：
1. 检测是否已有根约束（有则走修订 / 对齐漂移，不是重写）。
2. 按 C1-C6 探测前置材料（12 原则类硬约束、技术语境、流程 SOP、UED/安全等子约束）；**用户已给的清单优先于自动推断**。
3. 做挂载决策：哪些段落进根文档、哪些作为子约束被引用。
4. 草拟派生段落（触发路由表、冲突裁决链）。
5. 产出 **Phase 1 设计报告**（规范 §3.5 模板）→ **停在人工 checkpoint**，让用户拍板结构与开放问题，再进 Phase 2。

**Phase 2 · 生成**：
6. 按装配顺序生成各节（每节用规范 §4.2 的双子块小模板）。
7. 写入输出文件（默认项目根 `CLAUDE.md`，或用户指定名）。
8. 跑 §4.3/§4.4 内联自检；建议按 §5 做加载验证。

## 输出

一份根约束文档，含：优先级声明、硬约束（如 N 大原则）、§导航首跳指针、流程引用、**§触发路由表**、**§冲突裁决规则**、项目背景与子约束引用。结构按规范三形态之一，匹配项目复杂度。

## 自检红线（Phase 2 末尾，必跑 · 摘自 §4.4）

- 优先级链清晰、无环；冲突裁决覆盖「未列出的冲突 → 暂停问人」兜底。
- 触发路由表每行：任务特征 → 明确的必读子约束（路径真实存在）。
- 引用的子约束 / 技术语境文件路径可加载（无悬空引用）。
- 硬约束与子约束分层正确（硬约束进根文档，细节下沉子约束按需引用）。

## 可选 · 项目特性

> 本 skill 生成的是**通用根约束**。下列为 KDevSec 式项目的可选成分，按需取舍：12 条大原则作为硬约束、迭代→版本层级的 SOP 引用、产物元信息 + 上游依据@版本号 约定、UED / 安全子约束挂载。无相关前置材料则不强挂（规范 §1.2「按前置材料决定是否挂」）。
