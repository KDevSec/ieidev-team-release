# Agent 需求到设计规划规范 v2

本文是一份面向任意 AI Coding Agent 的独立规范，用于把需求文档或需求描述转化为详细设计制品。它不依赖任何特定框架、目录结构、命令系统或项目上下文。

## 1. 目标

Agent 应基于用户提供的需求输入，产出一组可支持后续任务拆解、实现、评审和验证的设计制品。

核心目标：

- 从需求中抽取业务目标、用户场景、范围、约束和不确定项。
- 优先从验收标准中提取可验证行为、字段、边界、异常、联动和非功能要求。
- 将可推断内容转化为明确设计假设。
- 将阻塞性未知转化为澄清问题。
- 将关键技术选择转化为带理由的设计决策。
- 产出架构、模块、数据模型、接口契约、测试策略、交付运行策略和风险登记。
- 在进入实现任务拆解前，检查设计是否满足约束、原则和质量门槛。

## 2. 输入契约

Agent 必须支持以下输入。只有“需求文档或需求描述”是必需项，其余均为可选项。

```markdown
# 设计规划输入

## 需求文档或需求描述（必需）

[描述要构建什么、为谁构建、解决什么问题、希望达到什么效果。可以是完整需求文档、用户故事、需求列表、验收标准或自然语言描述。]

Agent 必须优先分析需求中的验收标准，特别是以【Given】【When】【Then】形式描述的场景。验收标准是设计信息的主要来源之一，不应只阅读需求描述摘要。

需要从验收标准中提取：

- 业务动作：用户或系统在 When 中触发了什么。
- 前置条件：Given 中的角色、权限、数据、环境、状态。
- 期望结果：Then 中可观察、可验证的结果。
- 字段与属性：列表、详情、弹窗、表单、日志、接口返回中出现的字段、格式、默认值、空值表现和枚举值。
- 边界场景：数量上限、字段长度、空数据、分页、排序、默认值、状态边界。
- 异常场景：异常输入、越权访问、参数缺失、网络中断、接口超时、并发冲突。
- 联动行为：跨模块、日志、通知、搜索条件、状态刷新、策略下发等关联影响。
- 实际生效行为：涉及策略、权限、规则或自动化动作时，必须提取端侧或系统实际行为，而不只提取界面变化。
- 具体阈值：时间、数量、性能、重试次数、刷新周期、锁定时长等明确数值。

如果需求文档没有验收标准，Agent 必须在设计中标记“验收标准缺失”，并基于需求描述推导临时验收假设。
```

## 3. 执行原则

Agent 在生成设计制品时必须遵守以下原则。

### 3.1 需求优先

先理解需求，再做技术设计。不得在未分析目标用户、核心场景、验收标准和范围的情况下直接给出技术栈或架构结论。

### 3.2 验收标准优先

验收标准是最接近“可实现行为”和“可验证结果”的输入。Agent 必须从验收标准中反推：

- 需要哪些数据实体和字段。
- 需要哪些界面、接口或命令。
- 需要哪些权限和状态规则。
- 需要哪些边界与异常处理。
- 需要哪些测试用例和验收路径。

如果验收标准与需求描述存在冲突，应明确列出冲突并标记为待澄清。

### 3.3 区分事实、假设和未知

所有信息必须归类为：

- `事实`：用户明确提供的信息，包括需求描述、验收标准、技术约束和已有系统上下文。
- `假设`：Agent 基于上下文做出的合理默认。
- `待澄清`：不澄清会显著影响范围、架构、安全、成本或用户体验的问题。

### 3.4 阻塞问题有限化

Agent 不应把所有缺失信息都作为阻塞问题。只有满足以下条件的问题才标记为 `待澄清`：

- 影响核心功能范围。
- 影响关键用户体验。
- 影响数据、安全、隐私或合规。
- 影响主要架构方向。
- 多个合理解释会导向明显不同的设计。

建议最多提出 5 个阻塞性澄清问题。其他不确定项应以 `假设` 方式继续推进。

### 3.5 决策必须可追溯

关键技术选择必须包含：

- 选择了什么。
- 为什么这样选。
- 该选择带来的影响。
- 剩余风险。

### 3.6 设计先于任务

本规范的输出是设计制品，不是详细实现任务列表。不得在设计尚未稳定时直接生成编码任务。

### 3.7 简单性优先

默认选择满足需求的最简单设计。引入复杂架构、额外服务、异步流程、微服务、事件驱动、缓存、队列或复杂抽象时，必须说明必要性。

### 3.8 可验证性

设计必须包含测试策略和验收路径。每个关键模块、数据规则、接口契约或非功能要求都应有对应验证方式。

## 4. 标准执行流程

Agent 应按以下顺序执行。

### 步骤 1：需求解析

从输入中抽取：

- 产品或功能名称。
- 目标用户。
- 用户目标。
- 核心场景。
- 功能范围。
- 明确排除项。
- 验收标准中的可验证行为。
- 验收标准中的字段、枚举、格式、默认值和空值表现。
- 验收标准中的边界、异常、越权、环境异常和联动行为。
- 数据对象。
- 外部系统或第三方依赖。
- 非功能要求。
- 业务约束。
- 技术约束。

输出“需求理解摘要”。

### 步骤 2：信息分类

将关键信息分为：

- 事实
- 假设
- 待澄清

如果存在阻塞性待澄清问题，仍应继续产出“基于当前假设的草案设计”，但必须将整体状态标记为“草案，有阻塞项”。

### 步骤 3：技术上下文建立

建立技术上下文表，至少覆盖：

- 系统类型。
- 目标平台。
- 语言与运行时。
- 主要依赖。
- 存储方案。
- 外部集成。
- 接口风格。
- 认证与授权。
- 测试策略。
- 性能目标。
- 规模假设。
- 部署环境。
- 主要约束。

如果用户未提供技术栈，Agent 可以推荐默认技术栈，但必须说明理由。

### 步骤 4：设计质量门检查

在设计前执行质量门检查：

- 简单性：方案是否尽量简单？
- 可测试性：关键行为是否可测试？
- 安全与隐私：是否满足安全和隐私要求？
- 性能：是否满足性能和规模目标？
- 可维护性：是否便于维护和演进？
- 依赖控制：依赖是否必要且可控？
- 部署适配：是否适配部署和运维环境？
- 合规：是否满足合规要求，如适用？
- 验收可追溯性：设计是否能覆盖验收标准中的 Given、When、Then？

每个质量门输出：

```text
状态：通过 / 警告 / 失败
原因：
必要动作：
```

如果出现“失败”，设计必须提出修正方案或明确阻塞原因。

### 步骤 5：研究与决策记录

为以下内容生成设计决策记录：

- 技术栈选择。
- 存储选择。
- 架构风格。
- 关键第三方集成。
- 权限模型。
- 数据一致性策略。
- 性能策略。
- 部署策略。
- 高风险或高成本选择。

每条记录使用以下格式：

```markdown
### 决策：[主题]

**选择**：[选择]

**理由**：[为什么这样选]

**影响**：[对实现、成本、复杂度、风险的影响]

**剩余风险**：[仍需关注的风险]
```

### 步骤 6：设计制品生成

Agent 应生成以下设计制品。

#### 6.1 架构设计

包含：

- 系统边界。
- 高层模块或服务。
- 模块职责。
- 模块之间的数据流。
- 关键依赖关系。
- 目录结构建议，如适用。

#### 6.2 关键流程设计

包含：

- 主要用户流程。
- 关键业务流程。
- 异常流程。
- 状态流转。
- 外部系统交互流程。

#### 6.3 数据模型

包含：

- 实体。
- 字段。
- 关系。
- 唯一性规则。
- 校验规则。
- 生命周期。
- 状态转换。
- 数据保留和删除策略，如适用。

字段必须优先从验收标准 Then 中的字段展示、列表反馈、日志记录、接口返回和策略生效结果中提取。

#### 6.4 接口契约

根据项目类型选择适当契约：

- Web 或 API：端点、方法、请求、响应、错误、认证。
- 命令行：命令、参数、选项、退出码、标准输出、标准错误。
- 库或 SDK：公开方法、类型、异常、用法示例。
- 事件驱动：事件名称、载荷、生产者、消费者、投递语义。
- 界面：页面、组件、交互、状态、错误提示、可访问性行为。
- 数据管道：输入结构、输出结构、转换规则、失败处理。

如果项目没有外部接口，应明确说明“无外部接口”的理由。

#### 6.5 错误处理与边界情况

包含：

- 输入错误。
- 权限错误。
- 网络或第三方服务错误。
- 数据冲突。
- 并发问题。
- 超时和重试。
- 降级策略。

这些内容必须优先覆盖验收标准中列出的边界、异常、越权和环境异常场景。

#### 6.6 安全与隐私设计

包含：

- 身份认证。
- 授权模型。
- 数据最小化。
- 敏感数据处理。
- 加密要求。
- 审计日志。
- 常见攻击面。
- 权限边界。

#### 6.7 可观测性设计

包含：

- 日志。
- 指标。
- 追踪。
- 审计事件。
- 告警。
- 调试信息边界。

#### 6.8 测试策略

包含：

- 单元测试。
- 集成测试。
- 契约测试。
- 端到端测试。
- 性能测试。
- 安全测试。
- 回归测试。

测试策略必须能追溯到验收标准。每类关键验收场景都应有对应测试方式。

#### 6.9 交付与运行设计

包含：

- 构建方式。
- 配置方式。
- 部署方式。
- 环境变量。
- 数据迁移。
- 回滚策略。
- 运维注意事项。

### 步骤 7：设计后质量门检查

设计完成后再次检查：

- 是否仍有阻塞性澄清问题？
- 是否所有“失败”质量门已解决？
- 是否所有关键技术选择都有理由？
- 是否存在明显过度设计？
- 是否覆盖验收标准中的关键场景？
- 是否可以进入后续任务拆解阶段？

## 6. 输出质量要求

合格输出必须满足：

- 不跳过需求理解。
- 优先分析验收标准，特别是 Given、When、Then 中的条件、动作和结果。
- 从 Then 中提取字段、格式、枚举、默认值、空值表现和实际生效行为。
- 覆盖边界、异常、越权、环境异常和联动场景。
- 不把所有未知都推给用户。
- 明确区分事实、假设和阻塞问题。
- 每个关键技术选择都有理由、影响和剩余风险。
- 数据模型和接口契约足够支持后续任务拆解。
- 测试策略能追溯到验收标准。
- 风险登记包含缓解措施。
- 最后明确说明是否可以进入任务拆解阶段。

不合格输出包括：

- 只给技术栈列表，没有设计。
- 直接生成任务清单，没有架构和模型。
- 没有解释关键决策理由。
- 忽略验收标准。
- 忽略安全、测试、部署或错误处理。
- 把不存在的信息当作事实。
- 对阻塞问题没有标记。
