# 详细设计 / 设计规划

把已冻结的需求 + 原型，细化成一份**可直接支撑任务拆解与 TDD 编码**的详细设计：完整字段表、接口协议/参数、数据模型、关键流程、错误处理、安全与隐私、测试策略、交付与迁移。设计**先于**任务拆解；每个设计决策**可追溯**到需求或显式假设。

## 权威规范（必读）

完整的执行流程、8 条执行原则、标准 18 节输出模板都在 **`references/agent-design-planning-spec-v2.zh-CN.md`**。它是本 skill 的唯一结构权威——动笔前 Read 它，**结构严格照它的 §5 标准输出模板，不自创**。本文件只做导航。

规范要点定位：
- **§3 执行原则** —— 需求优先 / 验收标准优先 / 区分事实·假设·未知 / 阻塞问题有限化 / 决策可追溯 / 设计先于任务 / 简单性优先 / 可验证性。
- **§4 标准执行流程** —— 步骤 1 需求解析 → 2 信息分类（事实/假设/待澄清）→ 3 技术上下文建立 → 4 设计质量门 → 5 研究与决策记录 → 6 设计制品生成 → 7 设计后质量门。
- **§5 标准输出模板（18 节）** —— 摘要 / 需求理解 / 验收标准提取 / 事实假设待澄清 / 技术上下文 / 设计质量门 / 决策记录 / 架构设计 / 关键流程 / 数据模型 / 接口契约 / 错误处理 / 安全与隐私 / 可观测性 / 测试策略 / 交付与运行 / 风险登记 / 设计后质量门。

## 工作流

1. **解析需求 + 提验收标准**（步骤 1-2）：从 SR/AR/原型抽出目标、范围内/外、验收标准；把信息分成**事实 / 合理假设 / 待澄清**三类。阻塞性待澄清有限化（≤ 少数几个），其余用标注了的合理假设推进。
2. **建技术上下文**（步骤 3）：锁框架/语言/存储/约定；复用既有能力优先，新机制与框架核心解耦。
3. **过设计质量门**（步骤 4）：开写前自检需求是否够清。
4. **记决策**（步骤 5）：每个关键取舍写成一条决策（背景 / 选项 / 结论 / 理由），可追溯。
5. **生成设计制品**（步骤 6）：按 §5 的 18 节模板逐节填，详略对照规范示例；**完整字段表 / 接口契约 / 数据模型 / 关键流程**是重头。
6. **过设计后质量门**（步骤 7）：自检覆盖度与可编码性。

## 输出

一份详细设计文档，**严格遵循规范 §5 的 18 节结构**（不自创结构）；可直接支撑下一步任务拆解与 TDD。决策有追溯、假设有标注、待澄清已收敛。

## 自检红线（设计后质量门 · 摘自 §4 步骤 7）

- 结构 = 规范 18 节，无缺漏、无自创节。
- 每条验收标准在设计中可找到对应落点（§15 验收→测试映射）。
- 关键决策可追溯到需求或显式假设；无「凭空架构」。
- 字段表 / 接口契约具体到可编码（类型、必填、参数、错误码），无「待定」黑洞。
- 简单性：资深工程师不会觉得过度设计。

## 可选 · 项目流程集成

> 若你的流程没有迭代 / 版本 / 契约枢纽概念，跳过本节。

若像 KDevSec SOP 那样：
- **详细设计是版本级产物**，归档 `迭代N/vX-<版本名>/03-design/详细设计.md`；头部加 front-matter（`迭代 / 版本 / 类型: 详细设计 / 状态 / 版本号 / 上游依据(@版本号) / 日期`）。
- **引用迭代级总体设计作为契约枢纽**；若本设计敲定/改动了影响其他版本的共享契约，**回写总体设计并升版本号**。
- 跨版本只引用前序版本详细设计的**共享面**（复用的机制 / 外键 / 集成模式），不把前序当契约权威。
- 模板之外补一节「下一阶段（编码）输入物清单」。
- 完成后过**人工评审检查点**，覆盖与需求/原型/总体设计的一致性。
