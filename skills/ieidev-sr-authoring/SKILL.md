# 系统级需求（SR）撰写与评审

把模糊的原始诉求收敛成一份**结构化、可评审、可追溯**的系统级需求文档（SR）。SR 回答「这个系统/功能集要做什么、成功长什么样、边界在哪」，**刻意不下钻到字段级、接口级、技术选型**——那些下沉到 AR 与详细设计。

## 权威规范（必读）

完整模板、创建流程、评审线都在 **`references/SR-Create-SPEC.md`**。它是本 skill 的唯一结构权威——动笔前 Read 它，按它的流程走。本文件只是导航与要点提炼，细节以规范为准。

规范要点定位：
- **§3 SR Template** —— 文件路径 / frontmatter / ID 约定（§3.1）、Part A Brief（§3.2）、Part B SR-Items 轻模板（§3.3）。
- **§4 Creation Procedure** —— 4 个阶段：①吸收上下文 ②澄清问题（brainstorm 风格提问，别替用户拍板）③写 spec 主体 ④自检；以及 §4.6 红线（必须停下问用户的情形）。
- **§5 Review Checklist** —— 机械规则 MC-1..5 + LLM 红线 R1..7 + 评审报告模板 + 决策规则。

## 工作流

1. **吸收上下文**（规范 §4.2）：读齐输入（背景、原始需求、相关旧系统/参考）。缺关键输入 → 停下补齐，不硬写。
2. **澄清**（§4.3）：用 brainstorm 风格把真实动机 / 成功标准 / 硬约束 / 边界问清楚。**MUST NOT 把「老板原话」直接当需求**。歧义处列多种理解请用户裁。
3. **撰写**（§4.4）：按模板产出 Part A（Brief：背景目标、成功标准、横向约束/集成边界、Open Questions）+ Part B（SR-Items，每条带稳定 ID）。**核心红线：写 WHAT 与 WHY，全程不出现技术栈 / schema / 接口定义 / 实现细节。**
4. **自检 + 评审**（§4.5 / §5）：跑机械规则 MC-1..5 与红线 R1..7；需要留档时按 §5.3 出评审报告。不通过 → 改 → 重检。

## 输出

一份 SR 文档（spec.md），路径 / frontmatter / ID 严格按规范 §3.1。Part A + Part B 齐全；所有 Open Questions 在评审版清零或显式列出。

## 自检红线（提交评审前）

- 全文 WHAT/WHY，**零**技术选型 / schema / 接口 / 实现细节（最常见的违规）。
- 每条 SR-Item 有稳定唯一 ID；成功标准可验证（无「较快」「尽量」等模糊词）。
- 不确定项显式列出，不埋在描述里。
- 机械规则 MC-1..5、红线 R1..7 全过（见规范 §5）。

## 可选 · 项目流程集成

> 以下是「带迭代 / 版本概念的研发流程」可叠加的约定。**若你的流程没有这些概念，跳过本节**——上面的规范驱动撰写已可独立交付。

若你的流程像 KDevSec SOP 那样区分 迭代 / 版本：
- **SR 是迭代级产物**（一个迭代一份），归档到迭代规划目录（如 `迭代N/00-planning/SR需求.md`）。
- 头部加极简 front-matter：`迭代 / 类型: SR / 状态 / 版本号(vN.N，草稿起 v0.1) / 上游依据(带 @版本号) / 日期`。
- `上游依据` 钉每个上游产物的 `@版本号`，供下游做「版本指针比对」一致性校验。
- SR 初稿完成后过**人工评审检查点**才进入下游（拆版本 / 总体设计）。
